## compose

### コンテナの役割

| コンテナ          | 常時起動 | 接続先・役割                                                                                                          |
| ----------------- | -------- | --------------------------------------------------------------------------------------------------------------------- |
| `gateway`         | はい     | 80/443だけを外部公開するNginx。ドメインごとに本番またはステージングのapplicationへ転送し、証明書を読む。              |
| `application`     | はい     | 本番用。Reactのビルド成果物とHono APIを同一オリジンで提供する。`production-app` の名前でgatewayから到達できる。       |
| `database`        | はい     | 本番用PostgreSQL。`internal`ネットワークだけに接続し、外部へ公開しない。データは`production-db`ボリュームへ保存する。 |
| `migrate`         | いいえ   | 本番DBのスキーマ変更時だけ一度起動する。applicationと同じコミットSHAのmigrationイメージを使い、成功後に終了する。     |
| `application-stg` | はい     | ステージング用application。`staging-app` の名前でgatewayから到達できる。                                              |
| `database-stg`    | はい     | ステージング用PostgreSQL。データは`staging-db`ボリュームへ保存し、本番DBとは共有しない。                              |
| `migrate-stg`     | いいえ   | ステージングDBのスキーマ変更時だけ一度起動する。                                                                      |

- application・database・migrateはポートを外部公開しない。
- `migrate` と `migrate-stg` は `migration` profile に属する。通常の `docker compose up -d` では起動せず、CDが `docker compose run --rm` で明示的に実行する。
- `APPLICATION_IMAGE` と `MIGRATION_IMAGE` は、同じコミットSHAから作成する。例: `abc123` と `abc123-migration`。

## CI/CD

CIとCDは完全に分類する

### CI

ci専用のymlを作成

- ブランチに関わらず、プッシュが起きたら発動
- 失敗したタスクはマージ or pull requestを出せないようにする

### CD

環境変数の設定、マイグレーション、ビルド、デプロイを行う。

- 本番用とステージング用は分ける
- ビルドはGHCRを使用する
- VPSがGHCRからイメージを取得する際は、GitHub Actionsが実行中だけ発行する`GITHUB_TOKEN`を使う。恒久的なGHCR用Personal Access TokenはEnvironment Secretへ登録しない。
- Deployment Planの最新状態をGitHub Variablesへ記録するため、Repository Secret `REPOSITORY_VARIABLES_TOKEN` を使う。このFine-grained PATは対象リポジトリだけに限定し、Repository permissionsの`Variables: Read and write`だけを付与する。VPS・GHCR・アプリの環境変数には使わない。

| YAML                    | 役割                   | 実行すること                                                                                            |
| ----------------------- | ---------------------- | ------------------------------------------------------------------------------------------------------- |
| `inject-env.yml`        | 秘密情報の配置         | GitHub Environment SecretからVPS上の`.env.prod` / `.env.stg`を生成し、権限を600にする                   |
| `build-and-migrate.yml` | 成果物作成とDB更新     | SHAタグでGHCRへpushし、その同じSHA用のマイグレーションを実行する                                        |
| `prepare-database.yml`  | DBの起動と初回判定     | DBを起動してhealthyになるまで待機し、Drizzleのmigration履歴がない初回DBかを判定する                     |
| `deploy_for_stg.yml`    | ステージングの自動反映 | mainのCI成功後、変更種別に応じてビルド・migration・application切替を選択して実行する                    |
| `deploy_for_prod.yml`   | 本番の手動反映         | 手動開始をUI確認完了として記録し、最新Planのdigestと技術判定を変更せず、本番Environment承認後に適用する |

### CDの実行順序

アプリケーションイメージを先にビルドしてGHCRへ保存し、その**同じSHA**に対応するマイグレーションを実行してからapplicationを切り替える。

```text
1. CI（型チェック・テスト）に成功
2. application用・migration用イメージを同じSHAでビルドし、GHCRへpush
3. VPSへ対象環境の.envを注入（権限600）
4. DBを起動してhealthyになるまで待機し、DBスキーマ変更時または初回DBのときだけ同じSHAのmigrationを実行
5. 成功した場合だけapplicationを同じSHAへ更新
6. `/ping` 成功後、Deployment Planを`pending_verification`として保存
7. 人間が主要な画面操作を確認し、問題がなければ本番workflowを手動開始
8. 本番workflow内で確認者・確認日時を記録し、最新Planを検証
9. 本番Environment承認後に同じPlanを適用
```

先にイメージを作るのは、マイグレーション対象のコードを確定させるためである。DB変更だけが成功した後にビルドが失敗する状態を避けるため、`migration → build` の順にはしない。

初回に空のDBを用意した場合は、ステージングの手動実行で`force_deploy`を有効にする。通常の差分判定を通さず、DBを起動してDrizzleのmigration履歴を確認する。履歴がなければmigrationを実行し、その後applicationを起動する。すでに履歴があるDBでは、通常どおりスキーマ変更がある場合だけmigrationを実行する。

旧システムのバックアップ復元は、このCD経路に含めない。空のDBへmigrationを完了した後、内容を確認したデータ専用dumpを管理者が手順に従って復元する。

```sh
# ステージングDBのmigrationを一度だけ実行する
docker compose --env-file .env.stg -f compose.stg.yml --profile migration run --rm migrate-stg

# 本番DBのmigrationを一度だけ実行する
docker compose --env-file .env.prod -f compose.prod.yml --profile migration run --rm migrate
```

### 環境変数変更時の重要なルール

`env_file` の内容は、コンテナを**作成する時**に渡される。`.env.prod` / `.env.stg` を更新しただけでは、すでに動いているコンテナの環境変数は変わらない。

そのため、環境変数を注入するYAMLの成功後は、対象applicationを必ず再作成する。DBを不要に再作成しないため、`--no-deps` を付ける。

```sh
# ステージング: .env.stg の APPLICATION_IMAGE もComposeへ渡して再作成する
docker compose --env-file .env.stg -f compose.stg.yml pull application-stg
docker compose --env-file .env.stg -f compose.stg.yml up -d --no-deps --force-recreate application-stg

# 本番
docker compose --env-file .env.prod -f compose.prod.yml pull application
docker compose --env-file .env.prod -f compose.prod.yml up -d --no-deps --force-recreate application
```

特に `WEBAUTHN_RP_ID`、`WEBAUTHN_ORIGIN`、`SECRET_KEY`、`DATABASE_URL` を変更した場合は、この再作成を省略してはいけない。更新後は必ず `/ping` とパスキーのログイン操作を確認する。

### 変更種別ごとの判断表

| 変更したもの                                             | イメージのビルド | DBマイグレーション | 再作成・反映対象                    | 確認項目                              |
| -------------------------------------------------------- | ---------------- | ------------------ | ----------------------------------- | ------------------------------------- |
| フロント・バックエンドのアプリケーションコードのみ       | 必要             | 不要               | application                         | `/ping`、変更した画面・API            |
| `package.json` / `pnpm-lock.yaml` / `Dockerfile.prod`    | 必要             | 原則不要           | application                         | 起動ログ、`/ping`                     |
| `drizzle/` またはDBスキーマに関する変更                  | 必要             | 必要               | migration成功後にapplication        | `/ping`、該当機能、DBエラーがないこと |
| `.env.prod` / `.env.stg` の値だけの変更                  | 不要             | 不要               | 対象applicationを`--force-recreate` | `/ping`、変更した設定を使う機能       |
| `VITE_*` の値を変更                                      | 必要             | 不要               | application                         | ブラウザで実際に生成されるURL・画面   |
| `compose.prod.yml` / `compose.stg.yml` のapplication設定 | 原則不要         | 原則不要           | 変更したサービスを再作成            | `docker compose config`、`/ping`      |
| Nginx設定・証明書                                        | 不要             | 不要               | gatewayをreloadまたは再作成         | HTTPS、対象ドメイン、プロキシ先       |
| DBデータの手動補正だけ                                   | 不要             | 内容による         | 通常は不要                          | 対象データ、アプリ画面                |

`VITE_*` はフロントエンドをビルドした時点でJavaScriptへ埋め込まれる。したがって、`.env`を差し替えてapplicationを再作成するだけでは反映されない。現在の本番・ステージングは同一オリジンのAPIを使うため、`VITE_API_URL` を設定しない設計とする。

### 初回移行の手順

初回だけは、現在公開中の旧サービスと新しいGatewayが同じ80番・443番ポートを同時に使えない。そのため、次の順番で切り替える。applicationとDBは外部へポートを公開しないため、旧サービスを動かしたまま先に準備できる。

1. **GitHubのEnvironment Secretを登録する**
   `staging` と `production` の両方へ、VPS接続情報、DB情報、WebAuthn設定、トークン設定を登録する。実際に使う値をリポジトリへ書かずに、CDから安全に`.env.stg`・`.env.prod`を作るための設定である。

   これとは別に、Repository Actions Secretへ`REPOSITORY_VARIABLES_TOKEN`を1つ登録する。これはステージング候補・承認・本番適用済みの状態をGitHub Variablesへ記録する専用のFine-grained PATであり、`staging`・`production`のどちらにも登録しない。

2. **VPSへ新しい設定ファイルを配置し、共有ネットワークを作る**
   VPS上の配置先へComposeファイルと`nginx`フォルダを置く。次に、Gateway・本番application・ステージングapplicationが通信するための共有ネットワークを一度だけ作る。

   ```sh
   docker network create schedule-gateway
   ```

3. **旧サービスを止める前に、新しいDBとapplicationを起動する**
   本番とステージングの両方を起動する。この時点では80番・443番ポートを使わないので、旧サービスと共存できる。`production-app` と `staging-app` の名前でapplicationが共有ネットワークから見つかることを確認する。

4. **Nginxの設定を事前に確認する**
   証明書ファイルがVPS上にあること、`nginx -t`が成功すること、本番・ステージング両方のapplicationへ内部通信できることを確認する。どちらか片方でも見つからない状態でGatewayを起動すると、両方のサイトが利用できなくなる可能性がある。

5. **旧サービスを停止し、すぐに新Gatewayを起動する**
   ここで初めて80番・443番ポートが空くため、新Gatewayを起動できる。同じIPアドレスとポートを使う以上、この切替の瞬間だけは短い停止時間が発生する。

6. **公開後の動作を確認する**
   本番・ステージング両方でHTTPS接続、`/ping`、パスキーログインを確認する。applicationを更新するCDは、Gatewayが稼働中なら自動で`nginx -s reload`も行う。これにより、application再作成後の接続先IPをGatewayへ読み直させる。

### DB変更時の注意

本番で旧applicationが動いている間にマイグレーションを実行するため、通常の変更は「追加してから切り替える」互換性のある変更にする。列削除・型変更・大規模データ変換は、次のように複数回のリリースへ分割する。

```text
第1回: 新しい列・テーブルを追加し、新旧applicationの両方で動く状態にする
第2回: 新applicationへ切り替え、必要ならデータ移行する
第3回: 旧applicationを使わないことを確認してから、古い列・テーブルを削除する
```

マイグレーションが失敗した場合はapplicationを更新しない。DBを手作業で戻す前に、失敗したSQL・データ・バックアップを確認する。

### Deployment Planの状態

| 状態                   | 意味                                                                   |
| ---------------------- | ---------------------------------------------------------------------- |
| `pending_verification` | ステージング配置と自動ヘルスチェックは成功したが、UI・機能確認は未完了 |
| `approved`             | 人間によるステージング確認が完了し、現在の最新候補を本番で利用可能     |
| `superseded`           | より新しいステージング候補が作られ、本番では利用不可                   |
| `deployed`             | 同じPlanが本番へ適用済みで、再適用不可                                 |
| `expired`相当          | 作成から90日を超過した、またはArtifactの保持期限が切れ、利用不可       |

Plan本体はステージングworkflowのArtifactとして保存する。人間がステージングを確認して本番workflowを手動開始すると、その実行者・実行日時を加えたPlanを`production-deployment-plan` Artifactとして保存する。GitHub Variablesは最新Artifactのrun ID、SHA、状態だけを保持する。新候補が成功した時点で、それ以前の承認は`superseded`となる。旧版へ戻す場合も、対象コミットから新候補を作り直して再確認する。

本番workflowの手動開始を「ステージングのUI・主要機能を確認し、この最新候補を本番へ進める」という意思表示とする。本番workflowは、Plan・期限・現在のステージングSHA・GHCR上のdigest・未適用を検証する。production workflowではmigration・再作成の判定を変更せず、本番で再ビルドしない。

ステージング確認で問題が見つかった場合は本番workflowを開始しない。修正後の新しい候補がステージングへ成功すると、古い候補は`superseded`となる。

### フロー

main push
↓
SHAタグでGHCRへpush
↓
ステージングへ同じSHAをデプロイ
↓
ヘルスチェック成功
↓
STAGING_DEPLOYED_SHAを自動更新
↓
Deployment Planをpending_verificationとして保存
↓
人間がステージングのUI・主要機能を確認
↓
問題がなければproduction workflowを手動開始
↓
確認者・確認日時を記録し、最新Planを検証
↓
production Environmentで最終承認
↓
同一digestを本番へ反映しdeployedを記録

- **mainにプッシュされたら、ステージング環境に自動デプロイを行う**
- **ステージング確認で問題があればproduction workflowを開始しない**
- **本番反映はproduction workflowの手動開始とproduction Environment最終承認を必須にする**
